4.6. Explicación de ROS_DOMAIN_ID: Aislamiento de Red en ROS 2

En esta clase aprenderás cómo ROS 2 utiliza la variable de entorno ROS_DOMAIN_ID para evitar que diferentes sistemas robóticos interfieran entre sí cuando comparten la misma red física (Wi-Fi o Ethernet).

📚 Documentación oficial:
https://docs.ros.org/en/jazzy/Concepts/Intermediate/About-Domain-ID.html

1. El concepto central: Aislamiento de red

🎧 Analogía con walkie-talkies: Imagina que es como configurar el canal de un par de walkie-talkies: si todos están en el Canal 0, se interrumpirán mutuamente. Si cambias a tu equipo al Canal 42, podrán comunicarse sin interferencias.

Por defecto, ROS 2 utiliza DDS (Data Distribution Service) como columna vertebral para sus comunicaciones. DDS emplea el Domain ID (ID de Dominio) para crear redes lógicas aisladas encima de la red física.

  • Mismo Domain ID: Los nodos pueden descubrirse y enviarse mensajes libremente de forma automática.
  • Diferente Domain ID: Los nodos se ignoran por completo. Esto evita comandos accidentales (por ejemplo, que un compañero de clase mueva tu robot por error).
  • El valor por defecto: Todos los sistemas ROS 2 utilizan el Dominio 0 de forma nativa.

2. Por qué el rango está limitado de 0 a 232

DDS no solo usa el Domain ID como una simple etiqueta de texto; lo introduce en una fórmula matemática para calcular los puertos de red UDP específicos que se usarán para enviar los datos.

Las reglas de redes establecen que los puertos UDP son enteros de 16 bits sin signo, lo que significa que el número de puerto más alto posible es 65,535. Debido a que la matemática de DDS escala hacia arriba junto con el Domain ID, alcanzar ese techo de 65,535 limita el ID de Dominio máximo posible a 232.

3. El "Rango Seguro" (0 a 101) y los puertos efímeros

Aunque técnicamente se puede llegar hasta el 232, la documentación recomienda encarecidamente mantenerse entre 0 y 101. Esto se debe a los puertos efímeros del sistema operativo (puertos temporales que se asignan aleatoriamente a aplicaciones cotidianas como los navegadores web).

En Linux, por defecto, los puertos 32,768 a 60,999 están reservados para este tráfico temporal. Si tu Domain ID de ROS 2 calcula un puerto de red dentro de ese rango, una aplicación como Google Chrome podría adueñarse del puerto que tu robot necesita, provocando pérdidas de comunicación.

  • Los Domain IDs del 0 al 101 se asignan a puertos por debajo de 32,768 (Zona segura).
  • Los Domain IDs del 102 al 214 se asignan directamente dentro de la zona de peligro.
  • Los Domain IDs del 215 al 232 se asignan por encima de ella.

4. Restricciones de participantes (La letra pequeña)

Cada vez que lanzas un nuevo proceso de ROS 2 (como ejecutar un nodo de telemetría o un nodo de cámara), DDS crea un "Participante". Cada participante toma sus propios puertos UDP unicast dedicados para recibir información.

Como estos números de puerto se incrementan con cada proceso individual que abres, pueden ocurrir dos tipos de conflictos:

Desbordamiento entre dominios

Si abres más de 120 procesos de ROS 2 en el Dominio 1, los números de puerto seguirán subiendo hasta invadir exactamente los números de puerto reservados para el Dominio 2.

Colisiones efímeras

Si eliges un Domain ID que está justo en el límite de la zona segura —como el Dominio 101— tienes muy poco margen de maniobra. El proceso número 0 se ubica de forma segura justo debajo del límite efímero, pero para cuando lanzas tu proceso número 54, la matemática obliga al puerto a entrar en el rango mayor a 32,768, arriesgándose a un conflicto con el sistema operativo.

6. Fórmulas de Cálculo (DDSI-RTPS)

Los puertos UDP se calculan utilizando las siguientes fórmulas basadas en la especificación DDSI-RTPS:

Variables base:
• Port Base (Puerto Base): 7400
• Domain Gain (Ganancia de Dominio): 250
• Participant Gain (Ganancia de Participante): 2

Fórmulas:
• Discovery Multicast Port = 7400 + (250 × domainId) + 0
• User Multicast Port = 7400 + (250 × domainId) + 1
• Discovery Unicast Port = 7400 + (250 × domainId) + 10 + (2 × participantId)
• User Unicast Port = 7400 + (250 × domainId) + 11 + (2 × participantId)

7. Cómo establecer ROS_DOMAIN_ID

Opción 1: En la línea de comandos (temporal)

Para un solo comando:

export ROS_DOMAIN_ID=42
ros2 run my_py_pkg py_node

Opción 2: En ~/.bashrc (permanente)

Para que sea permanente en tu sesión, agrega al final de tu archivo ~/.bashrc:

export ROS_DOMAIN_ID=42

Luego recarga tu terminal:

source ~/.bashrc

Opción 3: Verificar el Domain ID actual

Para ver qué Domain ID está activo en tu sesión:

echo $ROS_DOMAIN_ID

8. Mejores prácticas

  • ✅ Usa Domain IDs del 0 al 101 siempre que sea posible (zona segura).
  • ✅ En ambientes compartidos (aula, laboratorio), asigna un Domain ID único a cada equipo.
  • ✅ Documenta qué Domain ID usa cada robot o sistema para evitar conflictos.
  • ✅ Si usas Domain IDs cercanos al 101, evita lanzar muchos nodos en paralelo.
  • ✅ Verifica con la calculadora anterior si tus puertos estarán en la zona segura.
  • ❌ Evita Domain IDs entre 102 y 214 (colisionan con puertos efímeros).
  • ❌ Evita cambiar el Domain ID mientras los nodos están corriendo; detén todo y reinicia.
